# Animation Library Comparison

Research for: AI-generated mobile motion graphics (1080x1920), where Claude generates animation code and users iteratively edit it.

---

## Libraries Evaluated

| Library | Paradigm | LLM-friendly | 9:16 native | Video export | Single-file artifact |
|---|---|---|---|---|---|
| Remotion | Declarative React/JSX | High | Yes | Yes (Lambda/local) | Yes (.tsx) |
| Motion Canvas | Imperative TS generators | Medium | Yes | Yes (FFmpeg) | Yes (scene .tsx) |
| Theatre.js | Timeline GUI + JSON | Low | CSS-only | No (third-party) | No (JS + JSON state) |
| Rive | Proprietary binary | None | Yes (artboard) | No | No (.riv binary) |
| Framer Motion | Declarative React (UI) | Medium | CSS-only | No | Yes (.tsx) |
| GSAP | Imperative JS timeline | Low-Medium | CSS/Canvas | No (Puppeteer hack) | Yes (.js) |
| Lottie/dotLottie | JSON keyframes | Very Low | Yes | No (player only) | Yes (.json) |

---

## Per-question findings

### Q1: LLM generatability from text prompt

**Remotion (High):** Each frame is a React component. Visual state is a pure function of the current frame number via `useCurrentFrame()`. `interpolate(frame, [0, 30], [0, 1])` maps time to value declaratively. An LLM can reason about "at frame 0 the element is invisible, at frame 30 it is fully visible" and write the exact code that expresses it. The JSX tree mirrors the visual hierarchy directly.

**Motion Canvas (Medium):** Uses TypeScript generators and a coroutine/scene-graph model. `yield* all(rect.position.x(200, 1), circle.opacity(0, 0.5))` requires understanding of the generator sequencing model. LLMs can generate this but reasoning about concurrent vs sequential animation is error-prone. "Make this faster" requires finding and adjusting duration arguments scattered across yield chains.

**Theatre.js (Low):** Split between JS object setup and a JSON state file (`theatre-0.json`) that stores keyframes from the GUI editor. The JSON format is complex and not meant to be hand-authored. LLMs would struggle to generate or edit the state JSON reliably.

**Rive (None):** Animations are created in the Rive Editor desktop GUI and exported as binary `.riv` files. There is no meaningful way for an LLM to generate or edit Rive animations as code.

**Framer Motion (Medium):** JSX-based and declarative, so the format is LLM-friendly. However, it is designed for UI micro-interactions, not long-form motion graphics sequences. No frame/time concept for complex multi-step choreography.

**GSAP (Low-Medium):** Imperative timeline API: `gsap.timeline().to(el, {x: 200, duration: 1}).to(el2, {opacity: 0, delay: 0.5})`. LLMs can write this but the temporal reasoning is harder than Remotion's frame-based model. No native video export.

**Lottie/dotLottie (Very Low):** The JSON format is generated by After Effects plugins or the Lottie editor — not hand-authored. Raw Lottie JSON (with bezier control points, layer transforms, shape data) is too complex and brittle for LLM generation from scratch.

### Q2: 1080x1920 (9:16) portrait format support

**Remotion:** Native first-class support. `<Composition width={1080} height={1920} fps={30} durationInFrames={300} />`. No hacks or workarounds needed.

**Motion Canvas:** Scene size is configured in the project: `makeScene2D(function*(view) {...})` with `size: {x: 1080, y: 1920}` in project config. Works but less obvious than Remotion.

**Theatre.js:** CSS-based sizing. Can set any size in the browser but video export is not native — the "video" is whatever the browser renders.

**Rive:** Artboard dimensions set in the Rive Editor. Can be 9:16.

**Framer Motion / GSAP / Lottie:** All support arbitrary dimensions in principle, but none have a native "render this to a 9:16 video" pipeline.

---

### Q3: In-browser preview vs server-side render

**Remotion:**
- Browser preview: Remotion Studio (local Vite dev server) — frame scrubbing, timeline, real-time React rendering. Excellent DX.
- Server-side render: `@remotion/renderer` renders to MP4/GIF locally via Chromium headless + FFmpeg. Also works in Node.js for on-demand rendering.
- Cloud render: Remotion Lambda (see Q6). Parallel chunk rendering on AWS Lambda.

**Motion Canvas:**
- Browser preview: Vite dev server with built-in timeline UI. Good DX.
- Export: FFmpeg WASM in-browser or Node.js-based rendering. Less mature than Remotion's render pipeline.

**Theatre.js:** Browser-only preview. No built-in video export; would require Puppeteer recording.

**Rive:** Browser WASM runtime for interactive playback. No video export.

**Framer Motion / GSAP:** Browser-only; video export requires Puppeteer/MediaRecorder hacks.

**Lottie:** `lottie-web` for playback; `puppeteer-lottie` or `lottie-to-gif` for export. Not production-grade for 1080p video.

### Q4: Code artifact shape

**Remotion — single `.tsx` file:**
```tsx
import { AbsoluteFill, useCurrentFrame, interpolate, spring } from "remotion";

const MyAnimation = () => {
  const frame = useCurrentFrame();
  const opacity = interpolate(frame, [0, 30], [0, 1], { extrapolateRight: "clamp" });
  return (
    <AbsoluteFill style={{ backgroundColor: "#000" }}>
      <div style={{ opacity, color: "#fff", fontSize: 72 }}>Hello</div>
    </AbsoluteFill>
  );
};

export const RemotionRoot = () => (
  <Composition
    id="MyAnimation"
    component={MyAnimation}
    width={1080}
    height={1920}
    fps={30}
    durationInFrames={90}
  />
);
```

This is the entire artifact. An LLM can generate or modify it in a single response. Props and style values are the only levers.

**Motion Canvas — `src/scenes/scene.tsx`:**
```tsx
import { makeScene2D, Rect, waitFor, all } from "@motion-canvas/2d";
import { createRef } from "@motion-canvas/core";

export default makeScene2D(function*(view) {
  const rect = createRef<Rect>();
  view.add(<Rect ref={rect} width={200} height={200} fill="#ff0" opacity={0} />);
  yield* rect().opacity(1, 1);
  yield* waitFor(0.5);
  yield* all(rect().scale(2, 0.5), rect().fill("#f00", 0.5));
});
```

Single file is possible but the generator pattern (`function*`, `yield*`) is less intuitive for LLMs than JSX + frame interpolation.

**Theatre.js:** Two files — a JS setup file and a `theatre-0.json` state file with keyframes from the editor. The JSON is not hand-authorable at scale.

**Rive:** Binary `.riv` + JS runtime code. Not a code artifact.

**Lottie:** A single `.json` file but typically 1000s of lines of generated JSON. Not authored by hand or LLM.

### Q5: Maintenance, TypeScript, ecosystem

**Remotion:** Actively maintained (Jonny Burger + Remotion team, ~22k GitHub stars as of 2025). Full TypeScript throughout. Rich ecosystem: `@remotion/player` (embed in React app), `@remotion/lambda`, `@remotion/media-utils`, `@remotion/noise`, `@remotion/paths`, `@remotion/shapes`, `@remotion/rive` (Rive playback in Remotion), `@remotion/lottie`. Excellent docs.

**Motion Canvas:** Active (Aarthificial + community, ~8k stars). Full TypeScript. Smaller ecosystem. Good for a solo or small team use case.

**Theatre.js:** Slower cadence since 2023. TypeScript. ~10k stars. Less active issue resolution.

**Rive:** Very active (Rive Inc.). TypeScript runtime. Large community for interactive app use cases. Not relevant for video generation.

**Framer Motion:** Very active (Matt Perry / Framer). TypeScript. Huge ecosystem. Wrong tool for this use case.

**GSAP:** Mature and actively maintained (GreenSock). TypeScript types available. Large ecosystem. Better for web UI animations than code-generated video.

**Lottie/dotLottie:** Active (LottieFiles + community). Good playback ecosystem. Not relevant for LLM generation.

### Q6: Remotion Lambda — cost model, cold start, complexity

**Architecture:** Remotion Lambda parallelizes rendering by splitting a composition into chunks (e.g., a 300-frame video split into 30 chunks of 10 frames each). Each chunk is rendered by a separate Lambda invocation. Results are stitched together via S3 and FFmpeg.

**Cost:**
- Lambda pricing: ~$0.0000166667 per GB-second (us-east-1, 2025).
- A 10-second 1080x1920 composition at 30fps with `concurrency=20` might consume ~500-2000 GB-seconds total.
- Rough estimate: $0.01–$0.05 per render for a 10-30 second video. Actual cost depends heavily on composition complexity (GPU-like operations in headless Chrome scale with DOM complexity).
- S3 data transfer adds marginal cost.
- At scale (1000 renders/day), expect $10–$50/day. Manageable for a SaaS with per-render billing or subscription model.

**Cold start:**
- Lambda cold starts: 2–5 seconds for the first invocation. Remotion pre-warms by deploying with `--memory-size` of 2048+ MB and keeping functions warm.
- With chunk parallelization, the cold start only affects wall-clock time for the first batch. A warm function pool reduces P99 latency.
- Remotion provides `simulateRenderInParallel` to tune concurrency vs. cost tradeoff.

**Complexity:**
- Requires AWS account, IAM role with Lambda/S3/CloudWatch permissions.
- Remotion CLI handles deployment: `npx remotion lambda functions deploy`.
- S3 bucket must be created and configured.
- Moderate setup cost (~1-2 days of infrastructure work), then low operational complexity.
- Alternative: run `@remotion/renderer` on a persistent server (EC2, Fly.io, Railway). Simpler setup, no parallelization, but fine for queued rendering at lower concurrency.

**Recommendation:** Start with server-side `renderMedia()` on a Bun/Node.js server. Add Lambda when rendering bottlenecks are measured. Don't over-engineer for v1.

### Q7: Motion Canvas vs Remotion for iterative LLM edits

**Remotion wins for LLM iteration.** The key reason: visual state is a pure function of the frame number. Every visual property at every point in time is expressed as a direct mathematical transformation of `frame`. This means:

- "Make this faster": change `durationInFrames` on the Composition, or multiply the interpolation range: `interpolate(frame, [0, 20], ...)` instead of `[0, 30]`.
- "Make it bouncy": replace `interpolate` with `spring({fps, frame, config: {damping: 8, stiffness: 120}})`. One line change, localized.
- "Make it fade in from the left": add `translateX: interpolate(frame, [0, 20], [-200, 0])`. Additive, non-destructive.
- "Add a delay": shift the interpolation range: `interpolate(frame, [15, 45], [0, 1])`.

Each edit maps to a localized, predictable code change. An LLM diff over a Remotion file is small and surgical.

**Motion Canvas iterative editing is harder:**
- "Make this faster" requires finding all the duration arguments in `yield*` calls and reducing them proportionally. If animations overlap via `yield* all(...)`, the math gets complex.
- "Make it bouncy" requires adding an easing function as a third argument — but knowing *which* yield to modify requires understanding the generator's execution order.
- Generator sequencing means an LLM must simulate execution order mentally before generating a correct edit. This is error-prone.

## Recommendation: Remotion

Remotion is the clear winner for this use case. Summary of why:

1. **LLM output format**: Declarative JSX + `interpolate`/`spring` is the most natural format for LLM generation. The frame-as-pure-function model means every visual property is an explicit, locatable expression.
2. **9:16 native**: `width={1080} height={1920}` on `<Composition>`. No workarounds.
3. **Preview + export**: Remotion Studio for browser preview, `renderMedia()` or Lambda for MP4 export. Both paths are first-class.
4. **Single `.tsx` artifact**: The entire animation is one file the LLM can read, modify, and return in full.
5. **Ecosystem**: Best-in-class for code-based video: `@remotion/player` for embedding previews in the web app, `@remotion/lambda` for cloud rendering.
6. **Iterative editing**: "make it faster/slower/bouncier/delayed" all map to localized, predictable code changes.
7. **TypeScript**: Full TS throughout. No `any` escape hatches in the core API.

**Remotion Lambda for v1:** Start with local `renderMedia()` or a Bun server running the renderer. Lambda is the right long-term path but adds AWS infrastructure overhead that is unnecessary until rendering becomes a bottleneck.

## Minimal Remotion skeleton for 1080x1920

This is the minimal template an LLM should produce for a new animation. It compiles and renders immediately.

```tsx
// Root.tsx — register compositions here
import { Composition } from "remotion";
import { MyScene } from "./MyScene";

export const RemotionRoot = () => (
  <Composition
    id="MyScene"
    component={MyScene}
    width={1080}
    height={1920}
    fps={30}
    durationInFrames={90} // 3 seconds
    defaultProps={{}}
  />
);
```

```tsx
// MyScene.tsx — the actual animation
import {
  AbsoluteFill,
  useCurrentFrame,
  useVideoConfig,
  interpolate,
  spring,
} from "remotion";

export const MyScene = () => {
  const frame = useCurrentFrame();
  const { fps } = useVideoConfig();

  // Fade in over first 20 frames
  const opacity = interpolate(frame, [0, 20], [0, 1], {
    extrapolateRight: "clamp",
  });

  // Slide up with spring physics
  const translateY = spring({
    fps,
    frame,
    config: { damping: 12, stiffness: 100 },
    from: 60,
    to: 0,
  });

  return (
    <AbsoluteFill
      style={{
        backgroundColor: "#0f0f0f",
        display: "flex",
        alignItems: "center",
        justifyContent: "center",
      }}
    >
      <div
        style={{
          opacity,
          transform: `translateY(${translateY}px)`,
          color: "#ffffff",
          fontSize: 80,
          fontWeight: 700,
          textAlign: "center",
          padding: "0 60px",
        }}
      >
        Your Text Here
      </div>
    </AbsoluteFill>
  );
};
```

**Key API surface for LLM generation:**
- `useCurrentFrame()` — current frame number (0-indexed)
- `useVideoConfig()` — `{ width, height, fps, durationInFrames }`
- `interpolate(frame, [inStart, inEnd], [outStart, outEnd], options)` — linear mapping with clamping
- `spring({ fps, frame, config, from, to })` — physics-based easing
- `<AbsoluteFill>` — full-size container (1080x1920), positions children relatively
- `<Sequence from={30} durationInFrames={60}>` — time-shift child components

**LLM prompt hint:** Tell the LLM the frame range and resolution upfront. "Write a Remotion component for a 90-frame (3s at 30fps) 1080x1920 animation where..." gives it all the constraints it needs.

## Rejected alternatives: brief rationale

- **Motion Canvas**: More powerful for complex choreography, but the generator/coroutine model is significantly less LLM-friendly for both generation and iteration. Worth reconsidering only if Remotion hits hard limits with complex multi-layer sequences.
- **Theatre.js**: GUI-first. Not suited for LLM-code-first workflow.
- **Rive**: Binary format. No LLM path.
- **Framer Motion**: UI interactions only. No video export.
- **GSAP**: Imperative and DOM-dependent. No native video pipeline.
- **Lottie**: Tool-generated JSON. Not LLM-generatable at scale.
