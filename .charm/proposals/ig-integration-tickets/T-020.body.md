## Motivation

The Rust IG backend (T-011) spawns each Bun stage as a subprocess and parses its
stdout, but two stage CLIs aren't subprocess-drivable as-is. `score.ts`'s
`import.meta.main` (score.ts:339-355) prints a human-readable table via
`formatTable` -- no machine-parseable JSON -- and `store.ts`'s `import.meta.main`
(store.ts:239-325) is a fixture smoke harness with no argv handling at all. This
ticket adds the missing CLI surface so Phase A can capture per-frame scores and
Phase B can invoke store on the user's chosen frame set.

This is the explicit "additive Bun-side change, only if unavoidable" exception in
PROJECT.md (Non-goals: "No rewrite of the Bun pipeline ... The only acceptable
Bun-side change is additive"). ADD a CLI path; do NOT change scoring math,
thresholds, the store layout, the `writeExtraction` logic, or any exported
function's behavior.

## Scope (owns `scripts/ig-pipeline/score.ts` + `store.ts` only)

- `score.ts`: add a `--json` flag to the existing `import.meta.main`. With the
  flag, print the full `ScoreResult` returned by `scoreFrames` as a single line of
  `JSON.stringify(result)` to stdout (this is the whole object: both `scored` and
  `kept` arrays, each `ScoredFrame` with path/index/sharpness/delta/entropy/score/
  kept/rejectReason). Without the flag, the human `formatTable` output stays the
  default so nothing existing breaks. Strip `--json` out of the args before
  `resolveFrameArgs` so it is never treated as a frame path. Diagnostics (the
  too-few-survivors warning, already on `console.error` via `onWarn`) stay on
  stderr so stdout is a single clean JSON line.
- `store.ts`: add a real argv-driven path to `import.meta.main` that resolves
  `ExtractionPaths` via `resolveExtractionPaths(outRoot, sourceId, {date})` and
  calls `writeExtraction(paths, results)` with an `ExtractionResults` object built
  from JSON the caller supplies. Print the resulting paths
  (`{ dir, briefJson, extractionMd }`) as a single JSON line to stdout. Keep the
  existing fixture smoke behavior reachable (e.g. gate it behind no-args or a
  `--smoke` flag) but do not let it run -- or print to stdout -- when the real CLI
  args are present.
- Document the exact argv contract for BOTH new modes in a comment at the top of
  the respective `import.meta.main` block, since T-011 spawns against it.
- Do NOT touch `package.json`, the scoring algorithm, the thresholds, the
  `writeExtraction` / `placeMediaArtifacts` logic, `lib/paths.ts`, or any other
  stage file. (`lib/paths.ts` and `types.ts` are imported read-only.)

## Recommended argv contracts (confirm with T-011)

- `score.ts`: `bun run score.ts <frames-dir | frame paths...> --json` -- `--json`
  position-independent, everything else exactly as the current CLI already parses.
- `store.ts`: `bun run store.ts --results <file|-> --out <projectRoot> --id
  <sourceId> [--date <YYYY-MM-DD>]`, where `--results` points at a JSON file (or
  `-` for stdin) holding the four-key `ExtractionResults` object
  (`{ download, clip, scoreResult, analyze }`). Output: one JSON line
  `{ "dir": "<abs>", "briefJson": "<abs>", "extractionMd": "<abs>" }`.
  These are recommendations -- whatever shape ships must be the one T-011 parses,
  so keep it stable and documented; T-011 should `depends_on` this ticket.

## Acceptance criteria

- [ ] `bun run score.ts <dir> --json` prints exactly one line of valid JSON that
      `JSON.parse`s back into a `ScoreResult` (both `scored` and `kept`); nothing
      else goes to stdout. Without `--json`, the human table + summary still print
      unchanged.
- [ ] The `--json` output preserves `delta: null` for the first frame and OMITS
      `rejectReason` on kept frames AND on survivors ranked outside the top
      MAX_KEPT_FRAMES -- it must NOT serialize the `formatTable` "below-top-N"
      display string as a reject reason (that value is not in the frozen
      `RejectReason` union; absence of the key is the contract).
- [ ] `bun run store.ts` with the real argv writes the extraction via
      `writeExtraction` and prints the resulting extraction dir as a single JSON
      line; the smoke fixture path does not execute and does not pollute stdout
      when real args are present. The argv contract is documented at the top of the
      block for T-011 to spawn against.
- [ ] `bun run tsc --noEmit` is strict-clean and `bun test` is still green (the
      offline suite, incl. `score.test.ts`) -- no behavior change to `scoreFrames`,
      `writeExtraction`, or any exported function, and no live/network calls added.
- [ ] No change to scoring math, thresholds, store layout/paths, or `package.json`.

## Edge cases / failure modes

- "below-top-N" trap: `formatTable` (score.ts:328) renders a synthetic
  "below-top-N" string for frames that passed the filters but ranked out. That is
  display-only. The JSON must reflect the actual object -- `kept: false` with NO
  `rejectReason` key -- not that string.
- Flag stripping: if `--json` is not removed before `resolveFrameArgs`, the single
  remaining real arg is still a dir, but a stray `--json` left in a multi-path
  invocation would be `resolve()`d into a bogus frame path. Strip it explicitly.
- `writeExtraction` MOVES media (`reconcileFile` uses `rename`, falling back to
  copy+rm on EXDEV) and PRUNES frames in `framesDir` not present in
  `scoreResult.scored`. So the JSON paths passed to store must point at the real
  on-disk frame/clip/source locations, and the `scoreResult.scored` array the
  caller supplies determines which frames survive pruning. The user's keep/reject
  override (Phase B) is expressed by T-011 in the `scoreResult` it passes -- store
  consumes it faithfully and needs no override logic of its own.
- stdout discipline: any human-readable confirmation in the new store mode must go
  to stderr; stdout is the machine-parseable JSON line only. Same for score.
- `--results -` (stdin) must read to EOF before parsing; a malformed/empty JSON
  blob should exit non-zero with a stderr message, not print partial stdout.
- Determinism: store's folder name embeds today's date via `resolveExtractionPaths`;
  expose `--date` so T-011 (and any test) can pin it rather than depending on wall
  clock.
