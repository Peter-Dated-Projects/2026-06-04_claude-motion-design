## Motivation

Existing workspaces only re-lay-out the same three singleton panels
(terminal/preview/editor). The IG workspace needs NEW panel types (extraction list,
IG video preview, frame grid, brief). This ticket extends the workspace/panel model
so an IG stage exists in the bottom workspace bar and can host IG-only panels, with
a default layout matching the proposal -- WITHOUT the panel components themselves
(those are separate tickets that fill the registered slots). This is the "Workspace
panel model extension" known-unknown in PROJECT.md; the goal here is the smallest
extension that lets downstream panel tickets drop real components into stable slot
ids.

## SCOPE CORRECTION (read before planning)

The original ticket said "add new PanelIds to the panel type union in
`src/types/index.ts`." That is wrong and would not compile:

- `PanelId` is defined in `src/components/PanelLayout/SplitLayout.tsx:15`
  (`export type PanelId = "terminal" | "editor" | "preview"`). `src/types/index.ts`
  has NO panel types at all -- it holds domain types (`Project`, `Message`,
  `ProjectFile`). The IG panel ids MUST be added to the union in `SplitLayout.tsx`;
  there is no other way to make them type-valid `PanelId`s that `WORKSPACE_DEFS`,
  `renderPanel`, and `panelTitles` all accept.
- The "Instagram icon" requirement forces two more files. `WorkspaceBar.tsx` holds
  `ICONS: Record<WorkspaceIconKey, ...>` (exhaustive), so adding an `instagram` key
  to `WorkspaceIconKey` in `workspaceStore.ts` forces a matching row in
  `WorkspaceBar.tsx`, and the actual `InstagramIcon` SVG component lives in
  `src/components/icons.tsx` alongside `EditIcon`/`LayersIcon`/`RocketIcon`.

The `touches` set has been corrected to reflect the files the change actually
requires. This is a correction, not a scope expansion -- the logical scope ("add IG
panel types + an IG workspace stage with placeholder hosts") is unchanged. Keeping
the old (wrong) touches would have forced the worker to edit `SplitLayout.tsx` and
`WorkspaceBar.tsx` outside its declared scope, defeating the daemon's
touch-based serialization and risking a collision with any concurrent ticket.

`src/types/index.ts` stays in `touches` only as an optional home for shared IG
domain-type stubs if the worker finds it cleaner to colocate them; the panel/event
domain shapes themselves belong to the event-schema and panel-component tickets, so
edits here should be minimal or none. Do not invent domain types speculatively.

## Acceptance criteria

- [ ] The bottom workspace bar shows a selectable IG stage (Instagram icon, label
      "IG"); selecting it renders the IG default layout (sidebar + preview +
      frame-grid + brief regions) with placeholders, and the regions match the
      proposal's arrangement (left sidebar; preview + frame-grid top; brief below).
- [ ] The four IG `PanelId`s exist in `SplitLayout.tsx` and are wired into
      `renderPanel` + `PANEL_TITLES` + `PANEL_ORDER` so panel-component tickets can
      drop real components against a stable slot id with no further plumbing.
- [ ] Switching `editing` <-> `ig` and back leaves the three singleton panels and
      their live sessions (PTY/preview/editor) intact -- no remount, no regression to
      show/hide, persistence, or layout.
- [ ] The `editing` workspace's show/hide menu still offers only terminal/editor/
      preview; the `ig` workspace's menu offers only the four IG ids (no cross-leak,
      since `availablePanels` scopes both ways).
- [ ] `tsc` / `vite build` typecheck clean (note `strict: true` in tsconfig --
      exhaustive `Record<PanelId|WorkspaceIconKey, ...>` maps will fail loudly if any
      key is missing; rely on that as a built-in completeness check).

## Edge cases / failure modes

- Persistence migration: `sanitizeLayouts` / `validActive` in `workspaceStore.ts`
  drop any persisted layout/active id whose workspace is not in `DEFS_BY_ID`. Adding
  the `ig` def is safe for existing users (their persisted `editing` layout is
  untouched; no `ig` override exists yet so `selectActiveLayout` falls back to the
  def's `defaultLayout`). Do NOT change the persisted state shape.
- `mountedIds` is MONOTONIC (SplitLayout.tsx:295-301): once a panel id appears in any
  rendered layout it stays mounted and merely detaches (host removed from DOM) when
  its slot disappears. So IG panels will NOT unmount when the user leaves the IG
  workspace -- they stay mounted-but-detached and retain state, exactly like the
  three singletons. The original note ("IG panels mount/unmount with the workspace")
  is therefore FALSE under the current SplitLayout. This is fine for placeholders,
  but downstream panel tickets MUST NOT assume a fresh mount on workspace re-entry
  (no teardown of any video element / fetch on leave; reset on project switch, which
  the panels own, same as today's panels). Document this in the slot contract.
- `renderPanel` returns `ReactNode` from a switch; with the new ids unhandled it
  would return `undefined` and `createPortal(undefined, host)` renders nothing.
  Always return an explicit placeholder for each IG id so an empty slot is visible
  and obviously a stub, not a blank.
- `addLeaf` in App.tsx docks a re-shown panel by `PANEL_ORDER` index; ensure the four
  IG ids sit AFTER the three singletons in `PANEL_ORDER` so editing-workspace docking
  is unchanged and IG docking is sensible.
- Icon fallback: `WorkspaceBar` already does `ICONS[ws.icon] ?? LayersIcon`, so a
  missing icon degrades gracefully -- but add the real `instagram` row anyway;
  criterion 1 calls for the Instagram glyph specifically.

## Slot contract (document for the panel-component tickets)

Define and write down (in code comments next to the IG ids and/or a short note the
panel tickets read): the four slot ids, that a panel ticket fills a slot by adding a
`case` to `renderPanel` returning its component, that panels are long-lived
(mounted-once, detached-when-hidden -- NOT remounted per workspace entry), and that
per-project reset is the panel's own responsibility. Keep the hosting mechanism small
(a switch case is sufficient -- a registry is not required at this size); this is the
"likely a small extension, not a rework" called out in PROJECT.md.
