## Motivation

The IG workspace needs a single Zustand store as the source of truth for the active
extraction: current phase (idle/running-A/awaiting-review/running-B/done/error),
the source + clip, the scored frames with keep/reject + user overrides, the brief,
and the list of past extractions. All IG panels read/write through it. Foundational
-- the panels depend on this.

## Read first

`.charm/PROJECT.md` (Two-phase flow, success criteria 2-6), `CLAUDE.md` (Zustand
store conventions in `src/store/`, the `listen("event://name")` pattern), and the
Bun `scripts/ig-pipeline/types.ts` (ScoredFrame/ScoreResult/Brief/DownloadResult/
ClipResult shapes) plus the EVENT PAYLOAD SCHEMA defined in the Rust IG backend
module ticket (T-025, `src-tauri/src/commands/ig_pipeline.rs`) -- mirror those
field names exactly to avoid drift.

Concrete shapes to mirror from `types.ts` (these are the field names the store/types
must reproduce, not new fields to invent):

- `ScoredFrame`: `path`, `index` (1-based), `sharpness`, `delta` (`number | null` --
  null on the first frame, no predecessor), `entropy`, `score`, `kept` (boolean),
  `rejectReason?` (present only when `kept` is false).
- `RejectReason`: union of exactly `"low_sharpness" | "insufficient_change"`. Keep it
  a string-literal union; do not collapse to `string`.
- `ScoreResult`: `scored` (all candidates, capture order) + `kept` (survivors,
  ranked best-first by `score`). The store's default kept set comes from `scored`
  filtered on `kept === true`; ranking order lives in `kept`.
- `Brief`: nested `motionLanguage { energy, rhythm, pacing, transitions, signature }`
  plus flat `motionTheme`, `colorMood`, `typographyMotion`, `applicationGuide`.
- `StageName`: `"download" | "clip" | "frames" | "score" | "analyze" | "store"`.
  `StageProgress`: `{ stage: StageName, progress: number (0..1), message?: string }`.
- `DownloadResult`/`ClipResult` fields relevant to the store: `sourcePath`, `id`,
  `sourceUrl` (`string | null`), `durationSeconds`; `clipPath`, `clippedDurationSeconds`,
  `wasClipped`.

## Scope (owns `src/store/igStore.ts` + `src/types/ig.ts` only)

- `src/types/ig.ts`: TS types mirroring the cross-boundary contract -- the stage-
  progress event, the per-frame score record (path, sharpness, delta, entropy, kept,
  rejectReason), the brief shape, and the extraction-list item. Re-derive from the
  Bun types/Rust schema; this file is the frontend's view of the contract.
- `src/store/igStore.ts`: Zustand store holding phase state machine, current
  extraction (source/clip paths, video duration, wasClipped), scored frames + a
  user-overridable keep set (toggle a frame's kept flag), brief, error, and the
  past-extractions list + active selection. Actions: startPhaseA(input), apply
  stage-progress + per-frame events, toggleFrameKept(index), the chosen kept-frame
  path list selector for Phase B, startPhaseB(), load brief, setActiveExtraction,
  reset. Keep `invoke`/`listen` wiring thin here or in a small effect; the store is
  state + reducers.

## Acceptance criteria

- [ ] `src/types/ig.ts` reproduces the contract types listed under "Read first"
      with identical field names and the literal `RejectReason` union; a top-of-file
      doc comment states it mirrors Bun `types.ts` + the T-025 event schema and must
      track them (no `any`, no `string` widening of `RejectReason`).
- [ ] Store models the phase state machine as a discriminated/literal `phase` field
      covering `idle | running-A | awaiting-review | running-B | done | error` (or
      equivalent names), with transitions only via the actions below -- never set
      arbitrarily.
- [ ] Exposes a `keptFramePaths` selector returning the user-chosen frame paths,
      defaulting to `scored.filter(f => f.kept)` and mutated by `toggleFrameKept`.
      This is the exact list threaded into Phase B's `analyze` call.
- [ ] Applying a per-frame score event populates `frames` with all metrics
      (`sharpness`, `delta`, `entropy`, `score`), `kept`, and `rejectReason`;
      `toggleFrameKept(index)` flips the override-kept flag WITHOUT discarding the
      original `rejectReason` (so a re-rejected frame still shows its reason, and a
      re-kept frame can be toggled back).
- [ ] `applyStageProgress` and the per-frame apply reducer are pure store actions
      callable in isolation (no Tauri dependency), so the contract is unit-testable
      without a running backend.
- [ ] Follows the existing store idiom in `src/store/` (plain `create<State>()`,
      explicit action signatures, manual `localStorage` helpers if anything persists
      -- no new persist middleware); `invoke`/`listen` wiring stays thin (a small
      effect or action), not spread through reducers.
- [ ] Builds under `tsc`/vite typecheck; no panel imports yet (panels land later).

## Edge cases / notes

- This ticket does NOT subscribe to Tauri events end-to-end (that final wiring is the
  T-031 integration ticket) -- but it should expose the apply-event reducers those
  subscriptions will call, so the contract is testable in isolation.
- Keep the past-extractions list shape minimal (id, date, sourceUrl, thumbnail/first
  frame path, dir) -- the sidebar ticket (T-029) renders it; population can be a stub
  action the integration ticket fills.
- Don't duplicate the projectStore; reference the active project for the output root
  rather than re-deriving it.
- `delta` is `null` for the first frame (no predecessor) -- the type must allow it and
  any UI-facing logic must not assume a number. Don't coerce it to 0.
- Zero kept frames is a legitimate Phase-A outcome (T-025 surfaces it). The store
  should represent an empty `keptFramePaths` cleanly so a later panel can block the
  Analyze button rather than spending the paid call on an empty set -- the store just
  exposes the empty list; it does not own the button-disable UI.
- Frame events may arrive progressively/out of order; index by 1-based `index` (or
  `path`) into the frames collection rather than assuming append order, so a late or
  duplicate event updates the right slot.
- Boundary serialization casing is the real drift risk: Bun `types.ts` is camelCase,
  so T-025's serde structs must serialize camelCase (`#[serde(rename_all =
  "camelCase")]`) for these types to match as-is. Mirror Bun's camelCase here and, if
  T-025 lands with snake_case payloads, flag it rather than silently widening types.
  Note: neither T-025 nor this ticket declares a `depends_on` on the other despite
  sharing the contract -- coordinate via COORDINATION.md / T-031 wiring; recommend the
  orchestrator treat T-025's schema as the authority if the two disagree.
- `toggleFrameKept` overrides the keep decision but is distinct from the scorer's
  original `kept`; if the store keeps both (original + override), the `keptFramePaths`
  selector reads the override and panels can still show "was rejected: <reason>".
