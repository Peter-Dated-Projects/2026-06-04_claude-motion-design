## Motivation

The left sidebar: a VS Code file-tree-style list of past extractions for the active
project, so the user can revisit a prior brief/frames/video without re-running the
pipeline. Completes the designed experience and makes extractions a durable library.

## Read first

`.charm/PROJECT.md` (Left sidebar bullet + success criterion 6 + the per-project
listing known-unknown), the proposal's 'Extraction List' UI section + Storage layout
(`extractions/<YYYY-MM-DD>_<sourceId>/` with extraction.md frontmatter), and the IG
store (T-012) for the past-extractions list + `setActiveExtraction`.

## Scope (owns `src/components/IGWorkspace/ExtractionListPanel.tsx` only)

- Read the past-extractions list from the IG store (T-012 exposes each item as
  `{ id, date, sourceUrl, thumbnail/firstFrame path, dir }`) plus the active
  extraction id. Render one row per item, labelled `date + source short id` (e.g.
  `2026-06-05 reel_abc123`); derive the short id from the item id/sourceUrl, do not
  invent a new field. Active row highlighted. Empty state when the list is empty:
  "Drop a URL or video to get started."
- Clicking a row calls the store's `setActiveExtraction(id)` to load that
  extraction's video/frames/brief into the main area (the store/integration hook
  performs the actual disk read; this panel only dispatches selection).
- Presentational + dispatch only; no Tauri wiring, no `invoke`/`listen`, no disk
  access in this component. Use the store's existing selectors/actions only.

## Acceptance criteria

- [ ] Lists past extractions from the store, one row each, labelled `date + source
      short id`; the active extraction's row is visibly highlighted.
- [ ] Renders the empty state ("Drop a URL or video to get started.") when the list
      is empty.
- [ ] Clicking a row dispatches `setActiveExtraction(id)` through the store; no other
      side effects.
- [ ] Long ids / urls truncate within the row (e.g. ellipsis) without breaking the
      sidebar width or wrapping into multiple lines.
- [ ] Component is pure presentational + dispatch: no `invoke`/`listen`, no Tauri/FS
      access, no `any` on the store-derived item type.
- [ ] tsc/vite typecheck clean.

## Edge cases / notes

- The actual enumeration of the project's `extractions/` dir + parsing each
  `extraction.md` frontmatter is owned by the store/integration hook + a backend
  command; this ticket consumes the resulting list. If the store exposes only a stub
  (empty or hardcoded) list for now, render against it -- the integration ticket
  populates it. Do NOT add disk-reading or parsing logic here to compensate.
- Long source ids / urls: truncate gracefully in the row (CSS ellipsis), keep the
  full value in a `title`/tooltip so it's recoverable on hover.
- Ordering: render the list in the order the store provides it; if the store does
  not guarantee an order, sort newest-first by `date` so the most recent extraction
  is at the top. Don't re-derive dates from disk.
- Missing thumbnail/firstFrame (older extraction, or run still finishing): fall back
  to a neutral placeholder or a text-only row; never crash on a null/absent path.
- Active-row during a live run: the in-progress extraction may not yet be a committed
  past-extraction item (Phase B's store stage finalizes it). Highlight based on the
  store's active id; do not assume every active extraction appears in this list.
- Keep styling consistent with the app's existing file-tree/sidebar look
  (`src/components/CodePanel/` file tree is the reference for row layout, hover, and
  active-row treatment).
