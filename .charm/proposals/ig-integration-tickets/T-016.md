## Motivation

The top-right main-area panel and the heart of the review step: shows every
candidate frame, marks the scorer's rejects, and lets the user override the keep set
before the paid analysis runs. This is what makes the two-phase flow worthwhile.

## Read first

`.charm/PROJECT.md` (Frame grid bullet + success criteria 2-3), the proposal's
'Extracted Frames' UI section, and the IG store (T-012) for the scored-frames list,
keep/reject + reason, and `toggleFrameKept(index)`. Also read
`src/components/CodePanel/AssetsView.tsx` -- it is the ONLY existing local-image
render path in the app, and (see "Image loading" below) it does not work the way
the original draft of this ticket assumed.

## Scope (owns `src/components/IGWorkspace/FrameGridPanel.tsx` only)

- Grid of frame thumbnails, 5px gap, zero-padded index label (001, 002...) beneath
  each. Scrollable. Frames appear progressively as the store (T-012) fills from
  Phase A per-frame events.
- Render the thumbnail from whatever renderable `src` the store exposes per frame
  (a data URI or asset-protocol URL -- see "Image loading"). This panel must NOT
  read files, call Tauri, or convert raw filesystem paths to image sources itself;
  it consumes a ready-to-render src field. If T-012's per-frame record only carries
  a raw `path`, that is a dependency gap (see notes) and the panel should fall back
  to a placeholder tile rather than wiring its own loader.
- Accepted frames: normal. Rejected frames: red semi-transparent overlay + centered
  X; hover tooltip with the reason ("Rejected: low sharpness" / "Rejected:
  insufficient change"). A user-overridden frame reflects its new keep state
  visually while still surfacing the original score/reason on hover.
- Click a frame to toggle keep/reject via the store's `toggleFrameKept(index)`
  (only in the awaiting-review phase; see edge cases). Presentational + dispatch
  only; no Tauri wiring, no backend calls.

## Acceptance criteria

- [ ] Renders all candidate frames with zero-padded index labels; rejects show the
      red semi-transparent overlay + centered X + reason tooltip; accepted show plain.
- [ ] Clicking a frame in the awaiting-review phase toggles its keep state through
      `toggleFrameKept`, the overlay/X updates immediately, and the original reject
      reason stays inspectable on hover after an override.
- [ ] Clicking outside the awaiting-review phase is a no-op (frames not interactive
      while Phase A is running or Phase B is analyzing).
- [ ] Grid fills progressively from store state (new frames appear as events land)
      and scrolls when candidates overflow the panel.
- [ ] Thumbnails render from the store-provided renderable src; if no src is
      available the tile shows a placeholder (no crash, no broken-image flash).
- [ ] tsc/vite typecheck clean; reads frames + reasons + phase from the store only
      (no Tauri/invoke, no filesystem access in this component).

## Edge cases / notes

- IMAGE LOADING -- this is the main risk and a likely dependency gap, not a thing
  this panel can solve alone. The original draft said to reuse AssetsView /
  `convertFileSrc`; both are inaccurate for this codebase:
    - `convertFileSrc` is used NOWHERE in the app, and the asset protocol is not
      enabled (`tauri.conf.json` security has no `assetProtocol`; capabilities grant
      only `fs:default`). So an `<img src={convertFileSrc(path)}>` will not load.
    - AssetsView's working mechanism is the backend returning base64 data URIs
      (`list_assets`/`add_asset` -> `AssetFile { name, dataUri }`), and those
      commands are scoped to the project's `assets/` dir. Extraction frames live in
      the extraction scratch/frames dir, NOT `assets/`, so AssetsView's commands do
      not cover them.
  Net: turning a frame's filesystem path into a renderable `src` requires either a
  new backend command (read frame file -> data URI, mirroring `add_asset`) or
  enabling the asset protocol + scope. Both touch `src-tauri`/config and are OUTSIDE
  this ticket's `touches` (FrameGridPanel.tsx only). RECOMMENDATION: the per-frame
  src must be supplied by a dependency -- ideally T-012's per-frame record carries a
  ready `src`/`dataUri` field populated by the Rust backend (T-013/integration), so
  this panel stays purely presentational. Coordinate this before implementing;
  flag it in `.charm/COORDINATION.md`. This panel should consume that field and not
  invent a loading scheme. (Not blocking this ticket -- the panel is correctly
  sized; this is a contract gap to resolve with the store/backend tickets.)
- Many candidates (up to ~60): keep thumbnails light (`loading="lazy"` as AssetsView
  does); lazy-render/virtualize only if trivially in reach, otherwise a plain scroll
  grid is fine for v1.
- Phase gating: read the phase from the store; toggling is only valid in
  awaiting-review. Outside it, render frames non-interactive (disabled cursor / no
  click handler) so the user cannot mutate the keep set mid-analysis.
- Progressive fill: frames arrive over time during Phase A; key tiles by a stable
  frame identity (index or path), not array position, so React reconciles correctly
  as the list grows and avoids remounting/flicker.
- Override semantics: `toggleFrameKept` flips `kept` but must not discard the
  original `rejectReason` (per T-012) -- a re-kept frame still shows why the scorer
  rejected it on hover. The panel relies on the store preserving that; just surface
  it.
