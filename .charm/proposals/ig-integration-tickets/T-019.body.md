## Motivation

This is the leaf that makes the IG workspace actually work: mount the four real
panels into their slots, subscribe to the backend Tauri events and route them into
the IG store via pure apply-event reducers, wire the two-phase `invoke` calls, and
prove the whole loop in the running app. It is the integration check on every other
IG ticket's contract -- the place where backend event payloads, store reducers, and
panel reads are confirmed to actually line up at runtime, not just at compile time.

## Scope

- `src/components/IGWorkspace/useIGPipeline.ts` (new): the single place that bridges
  Tauri <-> store. It `listen`s to the backend's stage-progress + per-frame events
  and forwards each into the store's pure apply-event reducers (it does NOT contain
  reducer logic). It exposes/triggers `startPhaseA`/`startPhaseB` which `invoke` the
  T-025 commands. It populates the sidebar's past-extractions list (via the new
  list-extractions command) and, on `setActiveExtraction`, loads the selected
  extraction's video/frames/brief from disk into the store. It owns listener
  teardown.
- `src/App.tsx`: replace the T-024 placeholder slots with the real panel components
  (T-027..T-030), and mount `useIGPipeline` for the IG workspace (mounted/unmounted
  with the workspace, consistent with T-024's non-singleton IG panels).
- `src-tauri/src/commands/ig_pipeline.rs` + `src-tauri/src/lib.rs`: add and register
  a command to LIST a project's past extractions (enumerate `extractions/` + parse
  each `extraction.md` frontmatter into the list-item shape). Confirmed missing
  today (no `list_extractions`/`extractions` symbol in `src-tauri/src/`), so this is
  required, not conditional. Frame thumbnails for the grid must also reach the
  webview as renderable srcs -- see the frame-rendering note; if that needs a new
  command, add it here too and register it.

## Acceptance criteria

- [ ] The IG workspace renders all four real panels (no placeholders); `useIGPipeline`
      is mounted for the IG workspace and routes backend stage-progress + per-frame
      events into the store's apply-event reducers (no reducer logic in the hook).
- [ ] End-to-end in-app on a real reel URL: Phase A shows live staged progress and a
      progressively-filling frame grid with red-X reject overlays + reason tooltips
      -> the run pauses in `awaiting-review` -> clicking a frame toggles keep/reject
      and that change is reflected in `keptFramePaths` -> Analyze (Phase B) runs on
      the chosen set, the brief renders, and "Use as Style Reference" switches to the
      main project workspace and lands the wrapped brief in the Claude terminal.
- [ ] Frame thumbnails actually render in the grid in the running app (not broken
      image tiles) -- the integration delivers a renderable `src` per frame; the
      asset protocol is NOT enabled (capabilities expose only `fs:default`), so this
      mirrors the `AssetsView`/`list_assets` data-URI approach rather than
      `convertFileSrc`.
- [ ] The sidebar lists the project's past extractions (from the new backend
      command), and clicking a row reloads that extraction's video/frames/brief
      without re-running the pipeline; the newly-completed run appears in the list.
- [ ] Boundary contract verified live: the per-frame event populates every metric
      (`sharpness`, `delta` incl. `null` first frame, `entropy`, `score`), `kept`,
      and `rejectReason` -- i.e. the Rust serde casing matches the camelCase TS
      types with no field dropping out as `undefined`.
- [ ] `tsc`/vite typecheck + `cargo check` (from `src-tauri/`) clean; the existing
      workspaces, the three singleton panels, and their show/hide + persistence are
      unregressed.
- [ ] The verification run is documented in the activity log: the reel URL used,
      whether the T-022 toolchain or a dev PATH fallback supplied bun/yt-dlp, and the
      observed result for each step above.

## Edge cases / notes

- Frame rendering is the highest-risk integration detail. The grid panel (T-028)
  consumes a ready-to-render `src` and explicitly refuses to read files or call
  Tauri itself. The app's only working local-image path inlines bytes as a base64
  data URI from a backend command (`projects.rs::AssetFile.dataUri`, rendered by
  `AssetsView`); the asset protocol / `convertFileSrc` is unused and unconfigured
  (`src-tauri/capabilities/default.json` has only `fs:default`). So this ticket must
  surface each frame to the store as a data URI (or wire a backend
  frame-thumbnail command that returns one) -- do NOT enable a new protocol or
  invent a path scheme. With up to ~60 frames, prefer thumbnail-sized bytes over
  full-resolution to keep the payload sane.
- Cross-boundary casing is the silent failure mode: Bun `types.ts` is camelCase, so
  T-025's serde structs must serialize camelCase. If a field arrives `undefined` at
  runtime, the bug is a snake_case payload, not the store -- fix/flag at T-025, do
  not widen the TS types to paper over it. Treat the T-025 schema as authoritative.
- Listener lifecycle: register `listen` on hook mount and `unlisten` on unmount
  (await the listen-promise's unlisten fn; guard with a `disposed` flag for the
  async gap) so re-entering the IG workspace doesn't double-subscribe and
  double-apply events.
- Zero kept frames after scoring is a legitimate Phase-A outcome (T-025 surfaces it,
  T-023 represents the empty `keptFramePaths`); confirm Analyze stays disabled
  rather than spending the paid call on an empty set. Worth exercising as a non-happy
  path even if the main run keeps frames.
- "Use as Style Reference" + the Analyze button are T-030's; the workspace switch
  uses `workspaceStore.setActiveWorkspace` and delivery uses the unchanged
  `terminal_input` PTY command. This ticket must not modify the PTY/terminal or
  re-implement the handoff -- only confirm it lands in the main workspace.
- `src/App.tsx` and `src-tauri/src/lib.rs` are shared with earlier IG tickets
  (T-024 owns App.tsx; T-026 owned lib.rs). This ticket depends on T-024 and T-026,
  so its edits to those files are serialized after them; keep the lib.rs diff purely
  additive (new command line[s] only) and the App.tsx change limited to swapping
  placeholders for real panels + mounting the hook.
- Verification needs the toolchain (T-022) present or a documented dev PATH fallback
  for bun/yt-dlp. Live analyze spends ~$0.21 -- one verification run is enough; reuse
  the proven public reel from the standalone pipeline's e2e rather than burning extra
  calls.
