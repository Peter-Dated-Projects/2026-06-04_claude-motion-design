## Motivation

The IG workspace runs the Bun pipeline as subprocesses, which needs the `bun`
runtime and the `yt-dlp` binary present. `claude` is already a hard app dependency,
so this ticket adds `bun` and `yt-dlp` to the existing on-demand toolchain
(download -> sha256-verify -> unpack) so a fresh install moves toward zero-config.

CORRECTION TO A PROJECT-LEVEL PREMISE (verified before scoping -- read this):
PROJECT.md and the original draft of this ticket claim "ffmpeg/ffprobe are already
bundled" in the render-toolchain. That is NOT true of the current toolchain. The
archive's `bin/` holds only `node` (build-render-toolchain.mjs only stages a node
binary + the Remotion render closure + render-mp4.mjs; render_toolchain.rs:71-78
`is_installed` checks only `node` + `render-mp4.mjs`). The `@remotion/*` deps it
installs include `compositor-darwin-arm64` -- that is Remotion v4's own Rust
compositor binary, NOT the `ffmpeg`/`ffprobe` CLIs. The Bun pipeline's clip/frames/
probe stages spawn bare `ffmpeg` and `ffprobe` off PATH (scripts/ig-pipeline/lib/
spawn.ts:117, `Bun.spawn([binary, ...args])`). So bundling only bun + yt-dlp leaves
the pipeline unable to clip/extract/probe, and success criterion 7 ("zero-config
run") will not actually hold. This ticket stays scoped to bun + yt-dlp as titled;
the ffmpeg/ffprobe gap is flagged in "Risks / open questions" for the orchestrator
to scope as a separate ticket (do NOT silently expand this one to cover it).

## Scope

- Extend `scripts/build-render-toolchain.mjs` to fetch `bun` and `yt-dlp` static
  binaries into the staged archive alongside the existing payload (the node binary +
  the Remotion render closure + render-mp4.mjs). Put them in the same `bin/` dir as
  `node` so one PATH entry covers all of them. Keep the script's existing
  darwin-arm64-only guard (it hard-aborts on other platforms today,
  build-render-toolchain.mjs:26-31); structure the fetch per-OS for later but do not
  attempt to make it cross-platform in this ticket. Print the new url/sha256/size as
  the script already does.
- Update the `TOOLCHAIN` manifest in `render_toolchain.rs` (new sha256/size after
  rebuild; see the version-bump note below) and add accessors that resolve the
  installed `bun` and `yt-dlp` absolute paths from the unpacked toolchain dir,
  modeled on the existing `node_bin()` / `render_script()` (render_toolchain.rs:63-69)
  -- there is NO ffmpeg accessor to mirror.
- ALSO expose a `toolchain_bin_dir(app)` accessor returning the unpacked `bin/`
  directory. This is the primitive the subprocess-spawning ticket actually needs:
  the Bun stages resolve `bun`/`yt-dlp`/`ffmpeg`/`ffprobe` by bare name off PATH
  (spawn.ts:117), so that ticket will prepend this dir to the child PATH rather than
  inject absolute paths. Per-binary accessors are still useful for presence checks.
- Add a presence/status signal the IG workspace can read WITHOUT changing MP4
  behavior: add a SEPARATE predicate (e.g. `ig_tools_installed(app)` checking the
  `bun` + `yt-dlp` files) and/or extend the `ToolchainStatus` struct with bun/yt-dlp
  presence fields. Do NOT fold the bun/yt-dlp check into the existing `is_installed`
  -- that predicate gates MP4 render, and adding the check would make any
  already-installed MP4 toolchain (or the dev fallback) read as not-installed,
  regressing MP4. Do NOT register new commands in lib.rs here (a separate
  registration ticket owns lib.rs); export whatever the command layer will call.

## Acceptance criteria

- [ ] The build script produces a darwin-arm64 archive whose `bin/` contains working
      `bun` and `yt-dlp` binaries (each runs `--version`), at pinned versions recorded
      in the script, and prints the new url/sha256/size.
- [ ] `render_toolchain.rs` exposes (a) accessors resolving the installed `bun` and
      `yt-dlp` absolute paths, (b) a `toolchain_bin_dir` accessor for the unpacked
      `bin/`, and (c) a presence predicate for bun + yt-dlp that is SEPARATE from
      `is_installed`.
- [ ] The manifest `version` (or whatever keys `toolchain_dir`) is bumped so existing
      installs re-fetch and pick up bun/yt-dlp; the presence predicate keys off the
      actual `bun`/`yt-dlp` files, so a stale toolchain dir lacking them reads as
      "IG tools missing" rather than falsely present. (See Risks: today `version` ==
      Remotion version, which would NOT change on a contents-only rebuild.)
- [ ] The existing MP4-render toolchain behavior is unchanged: `is_installed`,
      `render_toolchain_status`, the dev fallback, and `install_render_toolchain`
      still gate and run MP4 render exactly as before; an MP4-only toolchain (or the
      dev fallback) is NOT reported as broken by the new IG checks.
- [ ] No new commands registered in lib.rs from this ticket; exported symbols compile
      and are reachable by the (future) command layer.

## Edge cases / notes

- Bun and yt-dlp are single static binaries per OS; pin explicit versions and record
  them in the script. macOS arm64 is the dev target; the build script already
  hard-aborts on non-darwin-arm64, so this ticket is darwin-arm64-only by design.
- yt-dlp self-updates in the wild; pin a specific release for a reproducible sha256.
  Fetch yt-dlp's standalone single-file build (not the pip package) so it is one
  executable with no Python dependency.
- The two binaries are mach-o executables fetched from elsewhere (unlike `node`,
  which is copied from the running Node). After unpack, ensure they are chmod +x --
  extend the existing unix-permissions block (render_toolchain.rs:217-226) which
  currently only re-marks `node`. On macOS, downloaded binaries can carry the
  quarantine xattr; note whether Gatekeeper blocks them when spawned (the existing
  `node` is copied locally so never hit this) -- may need an xattr clear or ad-hoc
  codesign step, verify on a clean machine.
- The archive hosting/upload is the same out-of-band step the existing toolchain
  uses; if hosting isn't wired, leave the manifest url as the existing placeholder
  mechanism and document the rebuild step (don't block on hosting).
- Size budget: the current archive is ~77MB. Bun (~40MB) + yt-dlp (~30MB) roughly
  doubles it. Not a blocker, but update `size_mb` honestly so the install UI's
  download estimate stays truthful.

## Risks / open questions

- [BLOCKING for success criterion 7, NOT for this ticket] ffmpeg/ffprobe are NOT in
  the toolchain (see Motivation). bun + yt-dlp alone do not make the IG pipeline
  runnable: clip/frames/probe will fail with `binary_not_found` for `ffmpeg`/
  `ffprobe`. RECOMMENDATION to the orchestrator: add a sibling ticket to also vendor
  `ffmpeg` + `ffprobe` static builds into the same `bin/` (the `toolchain_bin_dir`
  primitive added here makes that a one-line PATH win for the spawn ticket), and
  extend the IG presence predicate to require all four. Kept out of THIS ticket to
  respect its stated bun+yt-dlp scope -- flagging rather than silently expanding.
- Version-key trap (also an acceptance criterion): `toolchain_dir` is keyed on
  `TOOLCHAIN.version`, which the build script derives from the Remotion version. A
  rebuild that only adds bun/yt-dlp would keep the same Remotion version, so existing
  installs (matching version dir + `.ok` marker) would never re-download and would
  silently lack the new tools. The fix is to bump the version (or otherwise
  invalidate the install) AND have the IG presence predicate check the actual files,
  not just `.ok`.
- License/redistribution: yt-dlp is Unlicense (fine to redistribute); Bun is MIT.
  No blocker, but confirm before hosting the combined archive publicly.
