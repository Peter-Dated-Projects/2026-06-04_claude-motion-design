## Motivation

The top-left main-area panel: where the user starts a run and watches Phase A
progress, then scrubs the clipped 30s result. It is the flow's entry point -- the
first thing a user touches in the IG workspace. It is purely presentational +
dispatch: it reads the IG store (T-023) for phase/progress/source/clip/error and
calls the store's `startPhaseA(input)`; it never wires Tauri events itself (the
store / a later integration ticket owns `invoke`/`listen`).

## Scope (owns `src/components/IGWorkspace/VideoPreviewPanel.tsx` only)

- **Empty state** (`idle`): URL text input + a native file drop zone. Enter in the
  input, a download/start button, or a dropped local video file all call the
  store's `startPhaseA(input)` -- passing the typed URL or the dropped file path.
- **Running state** (`running-A`, also `running-B` if a re-run streams here): a
  staged progress bar reading the store's current stage label + 0..1 progress
  (Downloading / Clipping / Extracting frames / Scoring), showing the source URL +
  detected duration. If duration > 30s, show the inline notice verbatim:
  "Analyzing first 30 seconds. Make sure your best moments are up front."
- **Done state** (`done`): a `<video>` player on the clipped 30s source (clip path
  from the store), scrubbable/replayable, replacing the input.
- **Error state** (`error`): a clear inline message sourced from the store's `error`
  field -- never a blank panel. Offer a way back to the empty state (e.g. a retry /
  dismiss that calls the store reset) if the store exposes one; otherwise just
  surface the message.
- Read all state from the IG store; do NOT call Tauri `invoke`/`listen` directly.
  This panel is presentational + dispatch only.

## Acceptance criteria

- [ ] Empty state accepts an Instagram URL (Enter or button) and a dropped local
      video file (native drag-drop, path hit-tested to this panel's rect) and both
      paths call `startPhaseA(input)` on the store with the correct input.
- [ ] The drop zone hit-test mirrors TerminalPanel: `devicePixelRatio`-scaled
      physical-pixel coords against this panel's `getBoundingClientRect()`, drop
      gated on the last enter/over hover (not the drop payload position), and the
      `onDragDropEvent` listener is cleaned up with a `disposed` guard on unmount.
- [ ] Progress bar + stage label track the store live during a run; duration > 30s
      shows the exact notice copy.
- [ ] Done state renders a working `<video>` player on the store's clip path
      (scrub + replay work).
- [ ] Error state renders the store's `error` message inline, never a blank panel.
- [ ] `tsc` / vite typecheck clean; no `any` for the store reads; no direct Tauri
      event wiring in this component.

## Edge cases / notes

- The native drag-drop event is window-global, so a drop on the frame-grid or
  sidebar must NOT trigger a run here -- the rect hit-test is what scopes it. Use
  the enter/over hover gate, not the drop event's own position (documented
  unreliable, especially with devtools attached).
- The drop payload carries `paths: string[]`; take the first path and don't assume
  a downloaded artifact -- the store / backend owns the local-file-vs-URL branch.
  This panel just forwards the raw path or URL string to `startPhaseA`.
- A dropped non-video file is the backend's / store's problem to validate; don't
  build a mime/extension allowlist here beyond an optional cheap guard, to avoid
  duplicating logic the store owns.
- Don't persist or derive phase/source/clip locally -- read them from the store so
  this panel and the rest of the IG workspace never diverge. Local component state
  should be limited to ephemeral UI (e.g. the in-progress text-input value and the
  drop-active highlight flag).
- Guard the `<video>` against a missing/not-yet-ready clip path (don't render a
  player with an empty src); fall back to the running/empty view until the store
  reports `done` with a clip path.
