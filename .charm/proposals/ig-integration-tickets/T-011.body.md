## Motivation

This is the backend engine for the IG workspace: a Rust module that spawns the
frozen Bun pipeline stages as subprocesses and streams their results to the
frontend as Tauri events. It implements the two-phase flow so the one paid analysis
call is gated behind user review.

## Read first

`.charm/PROJECT.md` (Backend invocation model + Two-phase flow + success criteria
2-4, 8), `CLAUDE.md` (frontend<->backend invoke/emit pattern; pty_bridge + render
toolchain as the reference for spawning + `emit` progress streaming), and the Bun
stage CLIs under `scripts/ig-pipeline/` (each `import.meta.main` prints its typed
JSON result; `types.ts` is the I/O contract; `download.ts <url> <outDir>`,
`clip.ts`, `frames.ts`, `score.ts`, `analyze.ts`, `store.ts`).

## (!) Blocking contract gaps found in review (resolve before implementing)

The scope below assumes "each Bun stage CLI prints its typed JSON result" and that
Phase B can "spawn analyze -> store". Reading the actual stage CLIs, two of the six
do NOT satisfy that, and both fixes land OUTSIDE this ticket's `touches`
(`scripts/ig-pipeline/`, which this ticket must not modify):

1. `score.ts` (`import.meta.main`, line ~339) prints a human-readable table via
   `formatTable(result)` + a summary line -- it never emits the `ScoreResult` as
   JSON and has no `--json` flag. The Rust module therefore has no JSON parse point
   for the score stage (the post-score per-frame event is the single most important
   payload this ticket produces). download/clip/frames/analyze DO print JSON; score
   is the lone exception.
2. `store.ts` (`import.meta.main`, line ~239) is a self-contained SMOKE HARNESS: it
   fabricates fake download/clip/score/analyze results and writes a throwaway
   extraction to a tmpdir. It does NOT parse argv. There is no arg-driven CLI that
   accepts the prior-stage results + kept-frame set and writes the real extraction.
   `writeExtraction(paths, results)` is exported as a function, but you cannot drive
   it as a subprocess as Phase B's scope requires.

Recommended resolution (additive, per PROJECT.md's "only additive Bun-side changes"
rule): a sibling ticket scoped to `scripts/ig-pipeline/score.ts` + `store.ts` adds a
`--json` result line to `score.ts` and a real argv-driven CLI to `store.ts` (accepting
the kept-frame list + an inputs payload, printing the resolved `ExtractionPaths.dir`
as JSON). T-025 should then `depends_on` that ticket. Without it, the Rust module
cannot complete Phase A's score parse or Phase B's store step within its own scope.
Flagging via `report_status(state="blocked")` so the orchestrator can sequence the
additive-CLI ticket ahead of this one rather than have the worker silently parse the
score table or shell out to store's smoke path.

## Scope (owns `src-tauri/src/commands/ig_pipeline.rs` only)

- Phase A command: given a URL-or-path + project output root, spawn (via the
  toolchain-resolved `bun` + `scripts/ig-pipeline/` scripts) download -> clip ->
  frames -> score IN SEQUENCE, parsing each stage's JSON stdout. Emit a Tauri event
  per stage (progress 0..1 + stage name) and, after score, an event carrying the
  full per-frame score/keep/reject set. Do NOT auto-run analyze.
  - Verified CLI invocations + outputs (the contract you parse against):
    - `bun download.ts <url-or-path> <outDir>` -> emits `STAGE <name> <0..1> [msg]`
      progress lines to stdout, then ONE final non-`STAGE ` line = `DownloadResult`
      JSON. Parse the last non-progress line, not the whole stream.
    - `bun clip.ts <sourcePath> <outDir>` -> `ClipResult` JSON (pretty-printed).
    - `bun frames.ts <clipPath> <outDir>` -> `FrameSet` JSON (pretty-printed).
    - `bun score.ts <frames-dir | frame paths...>` -> see blocking gap #1; today it
      prints a TABLE, not JSON. Treat the `--json` result line as the parse point
      ONLY once the sibling additive-CLI ticket lands.
  - `download.ts` is the only stage that interleaves `STAGE ` progress lines with its
    JSON; clip/frames/score emit no progress lines, so emit a single 0->1 stage event
    around their spawn rather than expecting incremental lines from them.
- Phase B command: given the chosen kept-frame path list (user may have overridden
  the keep set), spawn analyze -> store, emit progress, and return/emit the brief +
  final extraction folder path.
  - Verified: `bun analyze.ts <frame_path> [<frame_path> ...]` takes the kept-frame
    paths as positional argv and prints `AnalyzeResult` JSON (the override threads in
    directly, as PROJECT.md notes). store has NO usable CLI yet (blocking gap #2).
- Define the canonical Tauri EVENT PAYLOAD SCHEMA here (serde structs), mirroring the
  exact `types.ts` field names so the frontend (T-023) matches without drift. Pin
  these against `scripts/ig-pipeline/types.ts`:
  - StageProgress: `{ stage: StageName, progress: f64 (0..1), message?: String }`
    where `StageName` is the closed set `download|clip|frames|score|analyze|store`.
    Use `#[serde(rename_all = "camelCase")]` is unnecessary here (fields already
    single-word), but the StageName values MUST serialize lowercase to match.
  - ScoredFrame: `path, index (1-based), sharpness, delta (nullable -> Option<f64>,
    null for the first frame), entropy, score, kept (bool), rejectReason?` where
    rejectReason is the closed set `low_sharpness | insufficient_change`. delta and
    rejectReason being optional is load-bearing -- model them as `Option<_>` with
    `#[serde(skip_serializing_if = "Option::is_none")]` to mirror the optional TS keys.
  - The post-score event carries the full `ScoreResult` shape: `{ scored:
    ScoredFrame[], kept: ScoredFrame[] }` (scored = capture order, kept = ranked
    best-first) so the grid can render rejects + ordering from one payload.
  - Brief / AnalyzeResult for the Phase B result event: mirror `Brief`
    (motionLanguage.{energy,rhythm,pacing,transitions,signature}, motionTheme,
    colorMood, typographyMotion, applicationGuide) plus the extraction folder path.
  - These are camelCase in TS; if you name Rust fields snake_case, add
    `#[serde(rename_all = "camelCase")]` so the emitted JSON matches T-023's types.
- Resolve `bun` + the stage script paths from the toolchain (dev fallback to PATH +
  repo `scripts/` is acceptable, documented). Distinguish missing-tool vs stage
  ran-and-failed in the error surfaced to the frontend.
- Do NOT register the commands in lib.rs (a separate ticket owns the registry);
  expose `pub` command fns + a module-init/handler-list helper if useful.

## Acceptance criteria

- [ ] Phase A runs download->clip->frames->score by spawning the Bun CLIs and parses
      each stage's JSON (download: last non-`STAGE ` line; clip/frames: the JSON;
      score: the `--json` line once the additive-CLI ticket lands); emits per-stage
      progress events and a post-score event carrying the full `ScoreResult`; does not
      invoke analyze.
- [ ] Phase B runs analyze->store on an explicitly-passed kept-frame list (analyze
      takes them as positional argv) and yields the brief + final extraction folder
      path. (Blocked on store gaining a real CLI -- see blockers.)
- [ ] Event payload structs are defined with serde and serialize to the EXACT camelCase
      `types.ts` field names (StageProgress / ScoredFrame / ScoreResult / Brief), with
      `delta` and `rejectReason` as omitted-when-absent Options. No drift vs T-023.
- [ ] Zero-kept-frames is surfaced as a normal Phase-A result (empty `kept`), not an
      error.
- [ ] A failing stage surfaces a clear error NAMING the stage; a missing bun (or a
      download `SpawnError` for missing yt-dlp/ffmpeg) is distinguishable from a stage
      non-zero exit, mirroring render_toolchain's `TOOLCHAIN_MISSING` sentinel approach.
- [ ] `cargo check` passes; module compiles standalone (unregistered `pub` commands are
      fine as dead code until T-026 registers them in lib.rs).

## Edge cases / notes

- BLOCKERS first: the two contract gaps at the top (score has no JSON output, store
  has no real CLI) gate the score-parse and store steps. Do not work around them by
  scraping the score table or invoking store's smoke path -- both are brittle and the
  store smoke path writes to a tmpdir, not the project. Wait on the additive-CLI ticket.
- `download.ts` interleaves `STAGE <name> <0..1>` progress lines with its final JSON
  on one stdout stream; the JSON is the LAST non-`STAGE ` line. clip/frames/score/
  analyze emit only their JSON (no progress lines) -- emit a synthetic 0->1 stage event
  around their spawn.
- delta is `null` for the very first frame (no predecessor) -> `Option<f64>`, not 0.0.
  rejectReason is absent (not null) when `kept` is true.
- Path layout: `lib/paths.ts` owns `<out>/extractions/<YYYY-MM-DD>_<sourceId>/` and
  frame naming (`frame_NNN.jpg`, 1-based, zero-padded to 3). Do NOT re-derive any of
  this in Rust -- frames.ts/store.ts already write into it. Resolving paths has no FS
  side effects; only the store stage creates dirs. Recommend running stages into the
  final folder from the start and letting store finalize brief.json/extraction.md
  (PROJECT.md "known unknowns" leans this way), but document whichever you pick.
- Zero kept frames after scoring is a legitimate Phase-A outcome (`ScoreResult.kept`
  empty); surface it in the post-score event so the UI can block analyze rather than
  spending the ~$0.21 call on an empty set.
- Distinguish error classes in the surfaced message: (a) tool missing -- bun not at
  the toolchain path / PATH, or download.ts throwing `SpawnError` for a missing
  yt-dlp/ffmpeg (it catches `SpawnError` specifically and prints its message before
  exit 1); vs (b) a stage that ran and exited non-zero (name the stage). Mirror the
  `TOOLCHAIN_MISSING` sentinel convention from render_toolchain.rs so the frontend can
  branch to the install affordance instead of an error toast.
- Resolve `bun` from the toolchain dir (versioned, `{appData}/render-toolchain/...`)
  with a documented dev fallback to PATH + repo `scripts/`. yt-dlp/ffmpeg are the Bun
  stages' concern, not Rust's, beyond surfacing their failure.
- Use the same async-command + `app.emit(...)` pattern as render_toolchain.rs (which
  uses `tauri::{AppHandle, Emitter, Manager}` and streams `toolchain://progress`); do
  not block the main thread. Pick stable event channel names (e.g.
  `ig://stage-progress`, `ig://score`, `ig://brief`) and document them for T-023.
