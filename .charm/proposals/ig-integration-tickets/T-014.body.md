## Motivation

The IG backend module and the toolchain additions expose command functions but
deliberately do not touch `lib.rs` (to keep the registry a single-owner file and
avoid worker contention). This ticket is that single owner: declare the new module
and register every new `#[tauri::command]` in `generate_handler!`.

## Read first

`.charm/PROJECT.md` (Backend section + criterion 8), `CLAUDE.md` ("All commands are
registered in lib.rs's generate_handler! -- that list is the canonical index"), and
the two dependency tickets' modules:

- `src-tauri/src/commands/ig_pipeline.rs` (T-025 -- Phase A / Phase B `pub` command
  fns + the event-payload serde structs). T-025 explicitly does NOT register in
  lib.rs and may expose a handler-list helper; check what it actually exports.
- `src-tauri/src/commands/render_toolchain.rs` (T-022 -- bun/yt-dlp path resolvers +
  an IG presence/status signal). T-022 may EITHER extend the already-registered
  `render_toolchain_status` (then nothing new to register here) OR add an analogous
  status fn (then register it). Follow the module as built.

Concrete anchors in the current `lib.rs` (verified at this ticket's creation):
- The `commands` submodules are declared inline as `mod commands { pub mod ...; }`
  (no `mod.rs`) -- adding the backend module means inserting `pub mod ig_pipeline;`
  there, NOT a top-level `mod`/`use`.
- Existing commands are listed in `tauri::generate_handler![ ... ]` inside `run()`,
  one fully-qualified path per line (e.g. `commands::render_toolchain::render_toolchain_status`).
- Managed state is registered with chained `.manage(...)` calls before `.setup(...)`
  (currently `pty_bridge::PtyState::default()` and `file_watch::WatchState::default()`).

## Scope (owns `src-tauri/src/lib.rs` only)

- Add `pub mod ig_pipeline;` inside the inline `mod commands { ... }` block (only if
  T-025 did not already add it -- it owns a different file, so it almost certainly
  did not).
- Register the new Phase A, Phase B, and any NEW IG-toolchain-status command in
  `generate_handler!`, fully-qualified, in the existing one-per-line style.
- If the IG backend exposes managed state (e.g. a handle to an in-flight run / a
  cancellation token), register it via a `.manage(...)` chained call matching the
  existing pattern.
- Change nothing else in lib.rs (plugin setup, `.setup(...)`, existing command lines,
  the `greet` placeholder -- all untouched).

## Acceptance criteria

- [ ] `pub mod ig_pipeline;` is present in the `mod commands` block (added here only
      if T-025 did not).
- [ ] Every `pub` command fn from `ig_pipeline.rs` (Phase A + Phase B) appears in
      `generate_handler!`, fully-qualified, one per line.
- [ ] Any NEW toolchain status command added by T-022 is registered; if T-022 instead
      extended the existing `render_toolchain_status`, NO new toolchain line is added
      (and that is correct -- note it in the activity log).
- [ ] Any managed state the IG backend requires is registered via `.manage(...)`.
- [ ] `cargo check` (and `cargo build`) passes from `src-tauri/`.
- [ ] Diff against lib.rs is purely additive: no existing command line, plugin init,
      managed-state line, or `.setup(...)` body is altered or reordered.
- [ ] Registered command names exactly match the symbol names the frontend store /
      panel tickets `invoke(...)` (the Rust fn name is the invoke string).

## Edge cases / failure modes

- This ticket is the lib.rs contention point by design; depends on T-022 + T-025 so
  the symbols it registers exist. If spawned before both are complete, the referenced
  fns won't exist and `cargo check` will fail with unresolved-path errors -- that is
  the signal to wait, not to stub the missing fns. Block rather than invent symbols.
- Registering a command whose signature has a non-Tauri-serializable arg/return, or a
  fn not marked `#[tauri::command]`, fails at macro-expansion with an opaque error.
  Confirm each target fn carries `#[tauri::command]` in the module before adding it.
- Two ways to silently break the invoke contract: (a) a typo in the fully-qualified
  path, (b) registering a fn whose name differs from the frontend's invoke string.
  Cross-check names against the frontend tickets (T-023/T-027/T-028/T-029/T-030).
- If a command signature in the backend module changed during its build, the module
  is the source of truth -- register the actual signature, don't reconcile to a stale
  expectation in this ticket.
- Keep the registration block ordering stable/grouped (IG entries together, after the
  toolchain entries) to minimize merge noise with concurrent lib.rs-adjacent work.
