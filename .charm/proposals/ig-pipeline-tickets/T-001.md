Motivation: every pipeline stage shares a runtime, types, a subprocess helper, the output-folder layout, and the tuning constants. Build this foundation first so the six stage modules (Wave 1) can be written in parallel against fixed interfaces.

Authoritative context (read first, do not duplicate): `.charm/PROJECT.md` (scope, success criteria, non-goals), `.charm/proposals/PROPOSAL-instagram-pipeline.md` (full design), `.charm/proposals/SPIKE-instagram-pipeline.md` (verified commands, thresholds, costs). KB gotchas `claude-cli-no-image-flag` and `instagram-public-reel-no-cookies` and decision `0007-ig-pipeline-bun-runtime` are load-bearing.

Scope (this ticket owns the scaffold only, NOT any stage logic):
- `scripts/ig-pipeline/package.json` — Bun project; declare ALL anticipated deps up front so Wave-1 tickets never touch this file: an image lib for scoring (`sharp`), plus whatever the stages need. Add scripts: `test` -> `bun test`.
- `scripts/ig-pipeline/tsconfig.json`, `scripts/ig-pipeline/bunfig.toml` — standard Bun + strict TS config.
- `scripts/ig-pipeline/types.ts` — the stage I/O contract every module imports: e.g. `DownloadResult` (source path, id, source url, duration), `ClipResult` (clip path, original/clipped duration, was_clipped), `FrameSet` (clip path, ordered frame paths, fps, width), `ScoredFrame` (path, sharpness, delta, entropy, kept:boolean, rejectReason?), `ScoreResult` (all scored + kept ordered), `Brief` (the motion-language JSON schema from the proposal: motionLanguage{energy,rhythm,pacing,transitions,signature}, motionTheme, colorMood, typographyMotion, applicationGuide), `AnalyzeResult` (brief + costUsd + durationMs + numTurns), `ExtractionPaths` (the per-source output folder layout). Define a `StageProgress` type for structured `STAGE <name> <0..1>` stdout lines.
- `scripts/ig-pipeline/lib/spawn.ts` — a typed wrapper over `Bun.spawn` returning {code, stdout, stderr}; helpers to run a binary and capture/stream output, with a clear error when the binary is missing on PATH (yt-dlp/ffmpeg/ffprobe/claude).
- `scripts/ig-pipeline/lib/paths.ts` — resolve the per-source output folder and artifact filenames per the proposal's Storage section. Pin the layout exactly so T-007 (store) does not diverge: `<out>/extractions/<YYYY-MM-DD>_<sourceId>/` containing `source.mp4`, `clip.mp4`, `frames/` (subdir), `brief.json`, `extraction.md`. Frames are named `frame_NNN.jpg` (zero-padded, 1-based) — expose the frame-path builder here so frames.ts and store.ts share one naming source. The `extractions/` parent segment is part of the contract, not optional.
- `scripts/ig-pipeline/lib/constants.ts` — named, overridable tuning constants: FPS=2, SCALE_WIDTH=960, CLIP_SECONDS=30, DELTA_REJECT_THRESHOLD=8, SHARPNESS_FLOOR=50, MAX_KEPT_FRAMES (top-N), ANALYSIS_MODEL="claude-sonnet-4-6". No magic numbers inline in stage modules — they import these.
- `scripts/ig-pipeline/README.md` — how to run; external-binary prerequisites (yt-dlp, ffmpeg/ffprobe, claude) assumed on PATH; note this is CLI-only, no UI/Tauri this session.

Acceptance criteria (checklist):
- [ ] `bun install` succeeds in `scripts/ig-pipeline/` with no peer/engine warnings that would block CI.
- [ ] `types.ts` exports every interface above; `tsc --noEmit` (strict) type-checks clean with NO stage modules present — interfaces and lib helpers only, no stage impl.
- [ ] Every field a Wave-1 stage consumes is present on its input type and every field it produces is present on its output type, so the six stages compile against fixed contracts without amending `types.ts`. (`DownloadResult` -> `clip.ts`, `ClipResult` -> `frames.ts`, `FrameSet` -> `score.ts`, `ScoreResult` -> `analyze.ts`, `AnalyzeResult` + all prior -> `store.ts`.)
- [ ] `lib/spawn.ts` and `lib/paths.ts` are importable in isolation and have at least a smoke `bun test` proving they load and behave (path builder returns the pinned layout; spawn surfaces a typed "binary not found" error).
- [ ] `lib/constants.ts` values are overridable at runtime (env or param), not hardcoded `const` literals that stages can't override — the spike's thresholds (delta<8, sharpness<50) were one-reel samples and will need tuning.
- [ ] `README.md` documents the external-binary prerequisites, the per-stage CLI run pattern, and `bun test`; states CLI-only / no UI this session.
- [ ] `package.json` declares every anticipated dependency so no Wave-1 ticket has to touch it (scope-serialization: this file is owned here and frozen for the wave).

Edge cases / failure modes:
- spawn helper MUST distinguish "binary not found on PATH" from "binary ran and exited non-zero" (and ideally from a spawn/IO error) so stages can give actionable errors — a missing `yt-dlp` should not look like a download that failed.
- Keep the scaffold a separate runtime island from the existing Node `scripts/*.mjs` — do not import, re-export, or modify them; do not add ig-pipeline deps to the repo-root `package.json`.
- `sharp` is a native module: confirm it installs and loads under Bun on this platform during the scaffold (it is the load-bearing dep for T-005 scoring). If it does not load cleanly under Bun, that is a blocker to surface now, not a T-005 surprise — flag it rather than silently switching libs and breaking the scoring contract.
- Path builder must not assume the `<out>` root or the `extractions/`/`frames/` subdirs already exist; expose the resolved paths but leave mkdir to the store stage (or provide an explicit ensure-dir helper) so importing `paths.ts` has no filesystem side effects.
- `Brief` schema must match the proposal's JSON shape exactly (it is what `analyze.ts` parses out of the CLI `.result` envelope and what `store.ts` renders into `extraction.md`); a drift here breaks two downstream stages silently at runtime, not at compile time.
- Do NOT model a `--image` flag or any image-attach option anywhere in the spawn/types surface — per the load-bearing KB gotcha it does not exist; image analysis is Read-tool-over-paths only (T-006's concern, but keep the scaffold from implying otherwise).

Scope note: this ticket is scaffold + contracts only. Resist pulling any stage logic forward into `lib/` — if a helper feels stage-specific, it belongs in that stage's ticket, not here.
