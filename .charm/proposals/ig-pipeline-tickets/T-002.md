## Motivation

First stage of the pipeline: ingest a reel URL (or an already-local video) into a
working `source.mp4` for the rest of the chain. The KB gotcha
`instagram-public-reel-no-cookies` is load-bearing here -- public reels download
with NO auth on the first attempt (verified in the spike against
`https://www.instagram.com/reel/DYsvvtAhLUQ/`, yt-dlp 2026.03.17). Cookies are the
exception path, not a required onboarding step; building no-auth-first is what keeps
the common case zero-config.

## Scope (owns `scripts/ig-pipeline/download.ts` only)

- `download(input: string, outDir: string, opts?): Promise<DownloadResult>`.
  - If `input` is a local file path to an existing `.mp4`/`.mov`/`.webm`: skip
    download, derive an id, and use the file in place (do NOT copy/move it -- the
    next stage reads from its path).
  - If `input` is a URL: run yt-dlp via `lib/spawn` and write `source.mp4` into the
    per-source folder from `lib/paths.ts`.
- No-cookie first; on a login-required / age-gated / rate-limited error (detected
  from yt-dlp stderr/exit), retry ONCE with `--cookies-from-browser <browser>`
  (default `firefox`, overridable via `opts`/env). A transient network error is NOT a
  cookie trigger -- do not fall back blindly.
- Resolve the source id BEFORE building the output path (the folder name embeds it --
  see edge cases). Use yt-dlp's `%(id)s` for URLs; a cheap stable hash (e.g. of the
  absolute path, not the file bytes) for local files.
- Populate `DownloadResult.duration` from yt-dlp's reported metadata
  (`%(duration)s`) when available; for local input it may be undefined. `clip.ts`
  (ffprobe) is the authority on duration -- do NOT shell out to ffprobe here.
- Thin `import.meta.main` CLI: `bun run download.ts <url-or-path> [outDir]` prints the
  `DownloadResult` as JSON to stdout. Emit a `StageProgress` line (`STAGE download
  <0..1>`) so `run.ts` can compose it later.

## Acceptance criteria

- [ ] Running against the spike's public reel URL downloads `source.mp4` with no
      cookies and returns a populated `DownloadResult` (id, source path that exists on
      disk, source url, duration from metadata).
- [ ] A local-file input (`.mp4`/`.mov`/`.webm`) returns a `DownloadResult` pointing
      at the original path WITHOUT invoking yt-dlp at all.
- [ ] A local path that does not exist, or has an unsupported extension, fails with a
      clear actionable error (not a yt-dlp invocation).
- [ ] Missing yt-dlp binary produces a clear, actionable "not found on PATH" error
      via `lib/spawn` -- distinct from "yt-dlp ran and exited non-zero".
- [ ] The cookie fallback fires exactly once, and ONLY on an auth-signature error;
      the matched stderr substring(s)/exit condition are documented in a code comment.
- [ ] `tsc --noEmit` clean; the module imports `DownloadResult`/helpers/constants
      from T-001 rather than redefining them.

## Edge cases / failure modes

- Source-id chicken-and-egg: `lib/paths.ts` builds `<out>/<date>_<sourceId>/`, so the
  id is needed before the download target path exists. Resolve it with a metadata-only
  probe first (e.g. `yt-dlp --print "%(id)s" --skip-download <url>`), then download
  into the computed folder -- OR download with an `-o "%(id)s.%(ext)s"` template and
  derive the folder from what landed. Pick one; do not double-download.
- Cookie-fallback detection must key on the auth signature only. yt-dlp's
  login/rate-limit errors carry phrases like "login required", "rate-limit reached",
  "Restricted Video", "Sign in to confirm your age". Transient failures ("Unable to
  download webpage", HTTP 5xx, timeouts) must NOT trigger the cookie path -- match
  conservatively and document the exact substrings you key on.
- `--format mp4` worked in the spike but is fragile across reels; a robust fallback
  format chain (e.g. `bv*+ba/b[ext=mp4]/b`) is acceptable as long as the output ends
  up a single `.mp4`. Whatever you choose, `DownloadResult.source` must point at the
  file that actually landed (correct extension), not an assumed `source.mp4`.
- Do NOT bundle yt-dlp -- assume it is on PATH (documented prerequisite). Do not
  modify the existing Node `scripts/*.mjs`.
