Motivation: this is the bridge between a clipped video and the scoring stage. The pipeline needs an ordered, deduplicatable set of candidate stills sampled densely enough to catch fast text reveals and hold moments, but not so densely the scorer drowns in near-duplicates. The proposal fixes that sample rate at 2fps (60 candidates across a 30s cap). Satisfies success criterion 4 in `.charm/PROJECT.md`.

Read first: `.charm/PROJECT.md` and 'Stage 1 — Extract at 2fps' in `.charm/proposals/PROPOSAL-instagram-pipeline.md`. Import contracts/helpers/constants from T-001 (`types.ts`, `lib/spawn.ts`, `lib/paths.ts`, `lib/constants.ts`) — do not redefine any of them. Depends on T-001 landing first.

Scope (owns `scripts/ig-pipeline/frames.ts` only):
- `extractFrames(clipPath, outDir): Promise<FrameSet>`. Run `ffmpeg -i <clip> -vf "fps=<FPS>,scale=<SCALE_WIDTH>:-1" -q:v 2 frames/frame_%04d.jpg` (FPS, SCALE_WIDTH imported from `lib/constants` — no inline literals). Resolve the `frames/` subdir via the path helper from `lib/paths`, not by string-joining inline. Run ffmpeg through the `lib/spawn` wrapper so a missing binary vs a non-zero exit are distinguishable.
- Return a `FrameSet` exactly as typed in T-001's `types.ts` (clip path, ordered absolute frame paths, fps, width). Do not invent a new shape.
- After ffmpeg exits, enumerate the produced files from disk and sort by zero-padded index to build the ordered path list — do not synthesize the expected paths from a count (ffmpeg may produce fewer/more than predicted).
- Thin `import.meta.main` CLI: `bun run frames.ts <clipPath> [outDir]` prints the `FrameSet` JSON (or, for a terse mode, frame count + dir). CLI is a thin wrapper over `extractFrames`; no logic lives only in the CLI path.

Acceptance criteria:
- [ ] Given a clip, produces JPEGs at 2fps scaled to `SCALE_WIDTH` (960) width, preserving aspect ratio (`-1` height). A ~8.3s clip yields ~17 frames; a full 30s clip yields ~60.
- [ ] Returned frame paths are absolute, exist on disk, and are ordered by frame index (frame_0001 ... frame_NNNN).
- [ ] `FrameSet.fps` and `FrameSet.width` reflect the constants actually used.
- [ ] Re-running on the same `outDir` does not mix stale frames from a prior run with new ones.
- [ ] Missing `ffmpeg` on PATH produces a clear, actionable error (distinct from ffmpeg-ran-and-failed), surfaced via the `lib/spawn` error path.
- [ ] ffmpeg exiting non-zero (e.g. unreadable/corrupt clip) rejects with ffmpeg's stderr included, not a silent empty `FrameSet`.
- [ ] No constants hardcoded inline; all come from `lib/constants`.

Edge cases / failure modes:
- Stale frames: clear (or recreate) the `frames/` subdir before extracting so a shorter re-run doesn't leave higher-numbered frames from a previous longer run. Don't blow away sibling artifacts (source.mp4, clip.mp4, brief.json) — scope the delete to the frames dir only.
- Ordering: ffmpeg's `%04d` is zero-padded so lexical sort == numeric sort up to 9999 frames; rely on that but sort explicitly rather than trusting filesystem enumeration order.
- Frame count is approximate: ffmpeg's fps filter can emit one more/fewer frame than `ceil(duration*fps)` depending on rounding and the final partial second. Acceptance asserts a range ("~17"), not an exact count — do not hard-assert an exact number in code.
- Very short clips (<0.5s) may yield a single frame; still return a valid one-element `FrameSet`, not an error.
- `outDir` may not exist yet on a fresh run — ensure the frames dir is created before ffmpeg writes into it.
