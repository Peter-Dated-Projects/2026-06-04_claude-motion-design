Motivation: hard-cap any source to the first 30s and report metadata before frame work, so the rest of the pipeline always operates on a bounded `clip.mp4` and never silently drops a user's footage. This is Stage 0 (Probe + clip) in PROJECT.md and satisfies success criterion 3. It is the first consumer of T-001's `ClipResult` type, the `spawn` helper, and `CLIP_SECONDS` — proving those contracts end-to-end on real binaries.

Read first: `.charm/PROJECT.md` and the '30-Second Cap' + 'Stage 0 — Clip' sections of `.charm/proposals/PROPOSAL-instagram-pipeline.md`. Import contracts/helpers/constants from T-001 — do not redefine: `ClipResult` from `types.ts`, the subprocess wrapper from `lib/spawn.ts`, and `CLIP_SECONDS` from `lib/constants.ts`. The whole-clip output path (clip.mp4 inside the per-source folder) is owned by `lib/paths.ts`; if a helper for it exists there, use it rather than joining paths by hand.

Scope (owns `scripts/ig-pipeline/clip.ts` only):
- `clip(sourcePath, outDir): Promise<ClipResult>`. Probe via a single `ffprobe -v error -print_format json -show_format -show_streams <src>` call and pull duration/width/height/r_frame_rate from the first video stream. Then clip with `ffmpeg -y -i <src> -t <CLIP_SECONDS> -c copy <clip.mp4>` (stream copy = instant; `-y` so a re-run overwrites). Use `CLIP_SECONDS` from lib/constants — never hardcode 30.
- Sub-30s source: still produce clip.mp4 (fast copy), `was_clipped=false`. Over-30s: clip, `was_clipped=true`, report original vs clipped duration (so a UI can warn 'analyzing first 30s' later — no silent truncation).
- Return value must match T-001's `ClipResult` exactly (clip path, original duration, clipped duration, was_clipped, plus width/height/fps if the type carries them). Do not invent fields not in the type; if a needed field is missing, leave a note recommending it rather than diverging.
- Thin `import.meta.main` CLI: `bun run clip.ts <sourcePath> [outDir]` prints `ClipResult` as JSON to stdout (and only that JSON — diagnostics go to stderr so the output stays machine-parseable for `run.ts`).

Acceptance criteria:
- [ ] An 8s reel yields clip.mp4 roughly equal in length to the source, `was_clipped=false`, and a probed `duration` within a frame of the true 8s.
- [ ] A >30s input yields a clip.mp4 of ~`CLIP_SECONDS` and `was_clipped=true`, with BOTH the original and clipped durations present in the result (original > clipped).
- [ ] A source exactly at / within a hair of 30s does not get spuriously marked `was_clipped=true` (compare against `CLIP_SECONDS` with a small tolerance, not strict `>`).
- [ ] Returned object satisfies the `ClipResult` type with no `any`/cast escape hatches; the CLI prints valid JSON and nothing else on stdout.
- [ ] outDir is created if absent; an existing clip.mp4 is overwritten cleanly (no `ffmpeg` "already exists, overwrite?" prompt hang — that's what `-y` prevents).
- [ ] Missing `ffmpeg`/`ffprobe` on PATH -> clear actionable error naming the missing binary (distinct from a non-zero exit of a present binary), surfaced via T-001's spawn helper.

Edge cases / failure modes:
- `-c copy` cuts on the nearest keyframe, so a clipped duration can land slightly under/over `CLIP_SECONDS` — acceptable here; note it in a comment and don't assert exact equality in tests.
- Non-integer fps: `r_frame_rate` comes back as a rational string like `30000/1001`; parse it as numerator/denominator, don't `parseFloat` it.
- Duration may be absent or `"N/A"` on the video stream for some containers — fall back to `format.duration`, and error clearly if neither is present rather than emitting `NaN`.
- Rotated video: `width`/`height` from the stream are pre-rotation (a `displaymatrix` side-data / `rotation` tag may swap them visually). Report the raw stream dimensions for now; note the rotation caveat so a later stage can account for it.
- A source with no video stream (audio-only / corrupt) -> actionable error, not an index crash on `streams[videoIndex]`.
- `ffmpeg` may emit progress/warnings on stderr while still succeeding (exit 0); treat exit code, not stderr presence, as the success signal.
