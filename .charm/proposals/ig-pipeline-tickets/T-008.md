Motivation: chain all stages into one CLI command and prove the whole loop end-to-end. This is the leaf deliverable that closes success criteria 1, 7, and 8 in `.charm/PROJECT.md` — it is the only ticket that exercises every stage together, so it is also the integration check on the T-002..T-007 contracts. The spike (`.charm/proposals/SPIKE-instagram-pipeline.md`) already proved the manual five-stage chain works against the real reel and produced a correct brief; this ticket re-implements that chain as composable code and pins it with a test.

Read first: `.charm/PROJECT.md` (all success criteria), the 'Proposed Flow' diagram in `.charm/proposals/PROPOSAL-instagram-pipeline.md`, and `.charm/proposals/SPIKE-instagram-pipeline.md` (the reel URL `https://www.instagram.com/reel/DYsvvtAhLUQ/` is the e2e target; the spike's budget — ~41s, ~$0.21, 12 turns, 17->11 frames kept — is the reference for what a healthy run looks like). Import every stage's exported function (download/clip/frames/score/analyze/store, T-002..T-007) and the types/constants from T-001 — do not reimplement stage logic, re-derive thresholds, or duplicate the output-folder layout (that lives in `lib/paths.ts` from T-001).

Scope (owns `scripts/ig-pipeline/run.ts` and `scripts/ig-pipeline/run.e2e.test.ts` only):
- `run.ts`: `bun run run.ts <url-or-path> [outDir]` runs download -> clip -> frames -> score -> analyze -> store in sequence, threading each stage's typed result into the next (DownloadResult -> ClipResult -> FrameSet -> ScoreResult -> AnalyzeResult -> ExtractionPaths, all from T-001's `types.ts`). Resolve the output folder via `lib/paths.ts`; do not invent path logic here. Emit structured progress to stdout as `STAGE <name> <0..1>` lines (the `StageProgress` shape from T-001) so a later UI can parse them without a rewrite. Print a final summary (output folder, frames kept/rejected, brief motionTheme, costUsd). Any stage failure aborts non-zero with a clear, actionable message naming the failing stage.
- `run.e2e.test.ts`: an OPT-IN, env-gated smoke test (runs only when `IG_E2E=1`) that executes the full flow against the real reel URL and asserts the artifact set exists (source.mp4, clip.mp4, frames/ with >=1 frame, brief.json, extraction.md) and that the parsed brief is non-empty and schema-valid (matches the `Brief` type from T-001 — assert the shape, not the wording). Skips cleanly via `test.skipIf`/early-return (NOT a failure) when the env var is unset so `bun test` stays offline by default. This test hits the network and spends real money (~$0.21) when enabled — document that in a comment at the top of the file and in the README run notes if appropriate (README is T-001's; just call it out, don't edit it).

Acceptance criteria:
- [ ] `bun run scripts/ig-pipeline/run.ts <reel-url>` completes the full flow and writes the full artifact set (source.mp4, clip.mp4, frames/, brief.json, extraction.md) into the per-source folder resolved by `lib/paths.ts`.
- [ ] A local-file argument (a path to an existing .mp4) skips the download stage and runs clip -> store; a URL argument runs the full chain. (Honor the `<url-or-path>` contract in the command signature.)
- [ ] Each stage's typed result is threaded into the next via the T-001 types; no stage logic, thresholds, or path math is re-implemented in `run.ts`.
- [ ] Structured `STAGE <name> <0..1>` lines are emitted to stdout in stage order, ending each stage at 1.0.
- [ ] A stage failure aborts with a non-zero exit and a one-line, actionable message naming the failing stage (e.g. `download failed: yt-dlp not found on PATH`), not a raw stack-trace dump.
- [ ] Final summary prints output folder, kept/rejected frame counts, brief motionTheme, and costUsd.
- [ ] `IG_E2E=1 bun test` runs the e2e and passes against the real reel; plain `bun test` skips it, does zero network I/O, and spends no money.

Edge cases / failure modes:
- Keep the e2e tolerant of brief wording and frame count — assert schema/shape + non-empty, never exact text or an exact kept-frame number (the spike got 11, but that is reel- and threshold-dependent).
- The offline default `bun test` must do NO network and spend NO money — the skip must short-circuit before any stage runs, not inside a stage.
- Stage failures must surface the originating stage's error, not be swallowed or re-wrapped into a generic message; the abort message names which stage and why.
- A reel that yields zero kept frames after scoring is a legitimate failure case — `run.ts` should abort before calling analyze (don't send claude an empty frame list and burn cost) with a clear message.
- The local-path branch must not assume a download artifact exists; thread the provided path as the clip-stage input directly.
- e2e is slow (~40-60s for analyze alone) — give the test an adequate timeout so it doesn't flake on the real claude round-trip.

touches stays exactly `run.ts` + `run.e2e.test.ts`; this is already minimal and correct — no narrowing possible, no new file groups added.
