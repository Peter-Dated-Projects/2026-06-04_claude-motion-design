Motivation: persist the full extraction (inputs, frames, brief) in the per-source folder layout so a run is inspectable and reusable, and so a later UI/Tauri phase can list and reload past extractions without re-running the pipeline. This is the terminal stage that turns the in-memory stage results into the durable artifact set on disk. Storage section of `.charm/proposals/PROPOSAL-instagram-pipeline.md`.

Read first: `.charm/PROJECT.md` and the 'Storage' section of the proposal. Import the per-source folder layout and artifact filenames from T-001 `lib/paths.ts` (`ExtractionPaths`) and the stage I/O types from T-001 `types.ts` — do NOT redefine the folder layout, the filenames, or any stage result shape here. The proposal's canonical layout is:

```
extractions/<YYYY-MM-DD>_<sourceId>/
  source.mp4    <- original downloaded file
  clip.mp4      <- 30s clipped version
  frames/       <- all candidates (accepted + rejected), zero-padded
  brief.json    <- raw Brief JSON (the parsed .result from analyze)
  extraction.md <- human-readable brief + metadata frontmatter
```

Scope (owns `scripts/ig-pipeline/store.ts` only):
- `writeExtraction(paths: ExtractionPaths, results: { download: DownloadResult; clip: ClipResult; scoreResult: ScoreResult; analyze: AnalyzeResult }): Promise<void>` (decompose into focused helpers if it reads cleaner — `writeBriefJson`, `writeExtractionMd`, `placeMediaArtifacts`).
- Reconcile media into the per-source folder: ensure `source.mp4`, `clip.mp4`, and the `frames/` JPEGs end up at the paths `lib/paths.ts` resolves. The upstream stages may have written them elsewhere (e.g. a scratch/temp dir), so move or copy them into place; if a path is already the final location, no-op rather than failing.
- Write `brief.json` as the raw `Brief` (pretty-printed JSON, the parsed `.result` carried on `AnalyzeResult.brief`).
- Write `extraction.md`:
  - Frontmatter (every field sourced from a stage result, NOT hardcoded): `source` URL (from `DownloadResult`), `extracted` timestamp (ISO 8601, generated at write time), `originalDuration` + `clippedDuration` + `wasClipped` (from `ClipResult`), `framesAnalyzed` (count of kept frames sent to claude) + `framesRejected` (scored-but-rejected count, both from `ScoreResult`), `model` (from `AnalyzeResult`, e.g. `claude-sonnet-4-6`), `costUsd` + `durationMs` + `numTurns` (from `AnalyzeResult`).
  - Body: render the `Brief` as readable markdown — a `Motion Language` section with the five sub-fields (energy, rhythm, pacing, transitions, signature), then motionTheme, colorMood, typographyMotion, and applicationGuide. Do not dump raw JSON in the body; that is what brief.json is for.
- Thin `import.meta.main` CLI is optional (this stage composes from prior outputs); a small smoke entry that writes a sample `extraction.md` + `brief.json` from a fixture `Brief`/results into a temp dir is fine for standalone exercising.

Acceptance criteria:
- [ ] After a run, the per-source folder (resolved via `lib/paths`, not a literal path built in this module) contains `source.mp4`, `clip.mp4`, `frames/` (with the candidate JPEGs), `brief.json`, and `extraction.md`.
- [ ] `extraction.md` frontmatter is valid YAML and every field is populated from the stage results, not hardcoded; string values that can contain YAML-sensitive characters (the source URL especially) are quoted/escaped so the frontmatter always parses.
- [ ] `extraction.md` body renders all `Brief` fields (motionLanguage.{energy,rhythm,pacing,transitions,signature}, motionTheme, colorMood, typographyMotion, applicationGuide) as readable markdown — no missing fields, no raw JSON blob in the body.
- [ ] `brief.json` round-trips: parsing it back yields the same `Brief` object that was passed in.
- [ ] `framesAnalyzed` + `framesRejected` are consistent with the `ScoreResult` (analyzed = kept count, rejected = scored-but-not-kept; together they reconcile against the total candidate count).
- [ ] Re-running `writeExtraction` for the same source folder overwrites cleanly and leaves no stale or partial artifacts.

Edge cases / failure modes:
- Idempotent on re-run: overwrite `brief.json`/`extraction.md` and reconcile media in place; a second run must not duplicate, append, or error on already-present files.
- Don't assume `source.mp4`/`clip.mp4`/`frames/` already sit in the final folder — reconcile with `lib/paths`; handle both "already there" and "needs moving" without crashing.
- A `Brief` field may be missing or empty (e.g. `typographyMotion: "absent"`); render a placeholder rather than emitting broken markdown or throwing.
- `costUsd`/`durationMs`/`numTurns` may be absent on `AnalyzeResult` if the analyze stage couldn't parse them — emit the frontmatter key with a null/empty value rather than the string "undefined".
- Partial-write safety: if the run reaches Store, the brief is the valuable output — failing to move a media file should surface a clear error but ideally not lose the already-written brief.json/extraction.md.
